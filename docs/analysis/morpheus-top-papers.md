# Morpheus Top-Papers Analysis

> Research-lens synthesis of the Morpheus corpus (332 papers in `corpus/morpheus.db`
> plus 247 fresh arXiv entries in `corpus/staging-new.jsonl`) ranked against the
> Phase-E Morpheus design (`claude-mem/docs/morpheus-rl-design.md`) and the
> `morpheus-architecture.md` roadmap. Morpheus's problem shape: **learn a policy
> that scores, compresses, merges, or discards observation-memory chunks so that
> downstream retrieval injects the right ones into the next agent turn.** The
> training signal is trajectory-level feedback from a compiler that chose to
> retain, supersede, merge, or consolidate each observation.

---

## Executive summary

1. The literature has largely converged on Morpheus's shape: an agent maintains a
   **compact, fixed-size memory** that is **updated under an RL objective** whose
   reward comes from downstream task success. MEM1, MemAgent, MemSearcher, ReSum,
   RECON, and TeaRAG are all instances of this pattern — Morpheus is the same
   pattern applied to *observations-as-memory* instead of *document-chunks-as-memory*.
2. **GRPO-family algorithms dominate** (DeepSeek-R1, GiGPO, Kimi k1.5, DAPO-extensions)
   because they are critic-free and work well with verifiable rule-based rewards.
   Morpheus's signal schema (`retained / superseded / merged / consolidated`) is
   effectively a rule-based verifier emitting trajectory rewards — GRPO/DAPO is the
   natural fit for when Morpheus promotes beyond the weight-table scheme.
3. **Process-vs-outcome reward is the single most-studied axis** and it matters
   (StepSearch, MT-GRPO, ReasonRAG, IG-Search, CriticSearch). Morpheus's four
   signals are *intermediate process* rewards on a long trajectory — the field's
   empirical answer is clear: dense step-wise signal beats sparse outcome signal on
   long-horizon tasks, but only when the per-step reward has a verifiable or
   information-theoretic grounding (information gain, redundancy penalty, faithfulness).
4. For Morpheus's proposed **bucket-weight table**, the closest analog is PRM-style
   per-step classifiers (Math-Shepherd, Let's Verify Step-by-Step, PRMBench) — a
   deliberately simple tabular/shrinkage baseline is defensible because neural PRMs
   are known to collapse on small feedback corpora.
5. For **serving blend** (relevance × policy weight), IG-Search and Beyond-Correctness
   show multiplicative blending of retrieval score × learned faithfulness score is
   already standard; the design doc's proposal is well-calibrated.
6. **What's missing in the literature, and Morpheus owns it**: nobody trains a
   memory policy on the *compiler's retention decisions as reward* — every paper
   uses task-terminal correctness. Morpheus's `retained_in_context` signal is a
   genuinely new training signal.

## Tier S — Direct structural prior for Morpheus

### S1. MEM1 — `2506.15841` (2025)

- **Title:** MEM1: Learning to Synergize Memory and Reasoning for Efficient Long-Horizon Agents
- **Tags:** memory · algorithms-ppo-grpo · process-supervision · subject:planning
- **For Morpheus, this means:**
  - MEM1 *is* the structural analog of Morpheus: an agent keeps a **compact internal
    state** updated each turn via RL; the reward is *downstream task success*, not
    any explicit "memory quality" label. Morpheus's nightly-bucket update is the
    batch version of the same objective.
  - MEM1's "masked trajectory policy optimization" (`§3.2`) isolates gradient
    updates to agent-generated tokens only — directly applicable if Morpheus ever
    moves from bucket weights to a neural scorer, since we must not credit the
    compiler's injected observations.
  - MEM1 **consolidates rather than accumulates** prefix tokens into new internal
    state; its "dependency" metric is a principled way to measure whether
    Morpheus's consolidation is actually reducing long-range coupling.
  - Trained on 2-objective tasks, evaluated on N-objective tasks — supports
    Morpheus's thesis that policy learned on "did this obs get retained" will
    generalize across task types.
- **Key passages:**
  - > "MEM1 updates a compact shared internal state that jointly supports memory
    consolidation and reasoning."
  - > "prefix tokens from previous steps are consolidated into a new internal
    state, rather than being continuously accumulated."
  - > "gradient updates are localized to only tokens generated by the agent"
- **Open question:** MEM1 requires on-policy RL rollouts with a constant-memory
  enforcement. Morpheus is batch/offline — is there a MEM1-style masked update for
  our setting where the "agent actions" are historical `FeedbackRecorder` events
  and we cannot rollout counterfactuals?

### S2. MemAgent — `2507.02259` (2025)

- **Title:** MemAgent: Reshaping Long-Context LLM with Multi-Conv RL based Memory Agent
- **Tags:** memory · algorithms-ppo-grpo · method:grpo · subject:retrieval
- **For Morpheus, this means:**
  - MemAgent formalizes the **overwrite strategy** for a fixed-size memory: at each
    chunk, the memory is rewritten in full rather than appended. This is precisely
    the operation Morpheus's consolidation job performs on observation clusters.
  - Introduces **Multi-Conv DAPO** — a GRPO extension where each sample spawns
    multiple conversations and *the final-conversation reward is used to optimize
    all preceding conversations*. This is the algorithmic skeleton Morpheus needs
    if we want to credit earlier write-time decisions based on final retention.
  - The **reward design** is rule-based verifier — exact-match on ground-truth
    answers — and that's enough. Morpheus's `retained_in_context` is the analog of
    MemAgent's terminal correctness.
- **Key passages:**
  - > "MemAgent views an arbitrarily long document not as a monolithic block but as
    a controlled stream of evidence."
  - > "we adopt the RLVR recipe ... For the base algorithm, we adopt Group Relative
    Policy Optimization (GRPO)."
  - > "The answer contained in the final conversation is used to compute the reward
    and advantage, which are then employed to optimize all preceding conversations."
- **Open question:** MemAgent's memory is a span of tokens inside one context —
  Morpheus's memory is a database row joined on the bucket key. Can the Multi-Conv
  DAPO credit-assignment be re-expressed over bucket IDs so we can backpropagate
  (or just aggregate) retention from the compiler back to the *write-time bucket*
  without per-observation gradients?

### S3. MemSearcher — `2511.02805` (2025)

- **Title:** MemSearcher: Training LLMs to Reason, Search and Manage Memory via End-to-End Reinforcement Learning
- **Tags:** memory · algorithms-ppo-grpo · agentic-search
- **For Morpheus, this means:**
  - The most Morpheus-aligned paper in the corpus on a behavioral level: an RL
    agent explicitly decides **which turn content enters memory** and which is
    discarded. This is exactly the `retain/merge/supersede` decision Morpheus is
    trying to score post-hoc.
  - Uses end-to-end RL with a "keep only what's essential" loss — validates that
    terminal task reward is sufficient signal to train a keep/drop policy.
  - Trade-off framing ("preserving integrity vs. context cost") is precisely the
    trade-off Morpheus's policy-weight table is adjudicating via the 0.5–1.5 clamp.
- **Key passages:**
  - > "MemSearcher ... iteratively maintains a compact memory and combines the
    current turn with it."
  - > "update memory to retain only information essential for solving the task."
- **Open question:** MemSearcher rewrites memory at every turn. Morpheus's
  consolidation job runs nightly. How does update *cadence* interact with policy
  stability? Can we simulate frequent updates offline by re-running historical
  feedback streams in mini-batches?

### S4. GiGPO — `2505.10978` (2025)

- **Title:** Group-in-Group Policy Optimization for LLM Agent Training
- **Tags:** algorithms-ppo-grpo · method:grpo · process-supervision
- **For Morpheus, this means:**
  - Solves Morpheus's structural problem of **credit assignment over bucket keys**.
    GiGPO computes *two* advantages: episode-level (was the whole trajectory good?)
    and step-level (at matching anchor states, which action beat the group mean?).
    Bucket-keyed aggregation of retained/superseded signals across histories is the
    same idea: group all historical observations sharing a bucket key, then compute
    relative advantage vs. the global mean (matches Morpheus's §6 shrinkage-toward-
    global-mean update rule).
  - Critic-free, low-memory, stable convergence — matches Morpheus's constraint of
    never running gradient descent online.
  - Anchor-state grouping is **offline and hashmap-based** — Morpheus's bucket key
    *is* an anchor-state hash.
- **Key passages:**
  - > "Can we preserve the critic-free, low-memory, and stable convergence
    properties of group-based RL while introducing fine-grained credit assignment?"
  - > "this procedure incurs no extra rollouts: it is entirely offline and requires
    only lightweight key-based grouping using hashmaps."
- **Open question:** GiGPO uses a temporal discount γ on future rewards within a
  trajectory. Morpheus feedback arrives asynchronously (days later). What γ-analog
  handles wall-clock-delayed reward? Calendar-decayed reward weighting?

### S5. StepSearch — `2505.15107` (2025)

- **Title:** StepSearch: Igniting LLMs' Search Ability via Step-Wise Proximal Policy Optimization
- **Tags:** algorithms-ppo-grpo · process-supervision · method:ppo · retrieval
- **For Morpheus, this means:**
  - Introduces **information-gain reward + redundancy penalty** as two-component
    process reward. This is a better-principled alternative to Morpheus's
    `+1.0 / -0.3 / -0.1 / 0` table: we could compute IG over whether retrieving an
    observation *reduced downstream error* and penalize redundancy via the existing
    `merged_into` signal.
  - Dual-reward scheme (global outcome + local step) is structurally identical to
    Morpheus's planned blend of `retained_in_context` (step) + final task outcome
    (global, currently unused).
- **Key passages:**
  - > "Step-wise search involves interactive rounds, with information gain being
    rewarded and redundancy penalised."
  - > "We augment PPO with token-level rewards-information gain and redundancy
    penalties-for both query formulation and document retrieval."
- **Open question:** IG is computed against golden documents. Morpheus has no gold
  observation set — could we use the compiler's post-hoc choice as weak-gold, and
  accept the circular-reward warning in §8 of the design doc?

### S6. Process vs. Outcome Reward (ReasonRAG) — `2505.14069` (2025)

- **Title:** Process vs. Outcome Reward: Which Is Better for Agentic RAG Reinforcement Learning
- **Tags:** process-supervision · reward-models · algorithms-ppo-grpo · retrieval
- **For Morpheus, this means:**
  - Direct empirical answer to the "should we score steps or terminal outcomes"
    question that underlies Morpheus's entire §3 reward table. Result: process
    supervision wins on multi-hop tasks but only when per-step annotations are
    high-quality. Morpheus's four-signal table *is* process supervision; this paper
    validates the bet.
  - MCTS-based automatic process-reward annotation (SPRE, RAG-ProGuide) is a
    template for Morpheus if we later want to bootstrap synthetic feedback by
    having the compiler counterfactually replay retention decisions.
- **Key passages:**
  - > "We propose ReasonRAG, an automatic framework for agentic RAG process-level
    reward annotations."
  - > "MCTS guides the construction of the state-action tree and the assignment of
    process-level rewards for fine-grained policy optimization."
- **Open question:** ReasonRAG relies on counterfactual rollouts. Morpheus only
  has the trajectory that actually happened. Can shortest-path reward estimation
  be rephrased over the DAG of `consolidated_into` / `superseded_by` pointers?

### S7. Turn-Level Credit Assignment (MT-GRPO/MT-PPO) — `2505.11821` (2025)

- **Title:** Reinforcing Multi-Turn Reasoning in LLM Agents via Turn-Level Credit Assignment
- **Tags:** algorithms-ppo-grpo · process-supervision · method:grpo · method:ppo
- **For Morpheus, this means:**
  - First systematic study of turn-level reward design in multi-turn RL —
    **directly applicable to Morpheus's design question of whether to credit each
    signal separately or sum them.** The paper's answer: separate per-turn
    advantages, with PPO preferred over GRPO because GRPO needs exponential
    rollouts to compute intermediate advantages.
  - Strong caution for Morpheus: trajectory-level reward (what we currently have
    via summed signals) leads to "unstable training and suboptimal performance" in
    long-horizon settings. If the simple bucket-weight baseline underperforms, this
    is why — and MT-PPO is the escalation path.
- **Key passages:**
  - > "While multi-turn GRPO requires exponential rollout samples to compute
    intermediate advantages, multi-turn PPO leverages a critic model, offering a
    more efficient and scalable solution."
  - > "such coarse-grained credit assignment often leads to unstable training and
    suboptimal performance."
- **Open question:** For Morpheus, what's the analog of a "critic model" that
  scores the expected future retention of an observation given only its bucket
  features? A regression over historical bucket mean-retention could be the MVP.

### S8. ReSum — `2509.13313` (2025)

- **Title:** ReSum: Unlocking Long-Horizon Search Intelligence via Context Summarization
- **Tags:** memory · agentic-search · algorithms-ppo-grpo
- **For Morpheus, this means:**
  - ReSum is the **consolidation job** in miniature: a lightweight external tool
    periodically condenses interaction history into compact summaries, then the
    agent continues. Matches Morpheus's Phase-C Consolidation Job (#10) precisely.
  - Plug-and-play, no retraining required — validates the design decision to
    keep Morpheus's consolidator as a prompt-driven digest rather than a trained
    model.
- **Key passages:**
  - > "ReSum, a lightweight, plug-and-play paradigm that enables unbounded
    exploration by periodically invoking an external tool to condense interaction
    histories into compact summaries."
- **Open question:** ReSum's summaries are fed back into the same agent. Morpheus's
  consolidated digests are written to a different `is_consolidated_digest = 1` row.
  Do consolidated digests need a separate bucket namespace in the policy table to
  avoid cannibalizing their source observations' weights?

## Cross-cutting themes

1. **Process rewards beat outcome rewards on long-horizon tasks, but only when each step has a verifiable/measurable grounding** (StepSearch, MT-GRPO, ReasonRAG, IG-Search, `2505.14069`). Morpheus's four discrete signals are already such a grounding; the design is in-theme.
2. **Critic-free group-relative methods are the default for multi-turn agent RL** (GiGPO, DAPO, Multi-Conv DAPO, Kimi k1.5). If Morpheus ever escalates beyond weight tables, **GiGPO with bucket-key anchors is the natural next step** — no extra rollouts, purely offline post-hoc aggregation.
3. **Compact-memory + RL-trained-updates is a saturated pattern** (MEM1, MemAgent, MemSearcher, ReSum, RECON, TeaRAG, HiGMem). Morpheus is *applying* this pattern to its own claim domain (observation-level persistent memory, not document chunks) — differentiation is in the *reward source*, not the architecture.
4. **Shrinkage toward a prior is universal** when sample counts are small — PRMs use LLM-as-judge priors, bucket weights use global-mean shrinkage. The design doc's `weight = 1.0 + k * (μ - μ_global) / (1 + sample_count / 50)` is standard and correct.
5. **Serving-time reward/weight blending is multiplicative, not additive** in every paper that discusses it (IG-Search, Beyond-Correctness, StepSearch, TeaRAG). §10 q4 should be resolved in favor of multiplication.

## Gaps in the literature (Morpheus's novelty surface)

1. **No paper trains a memory policy on the compiler's retention decisions as reward.** Every RL-memory paper above uses *terminal task correctness* as reward. Morpheus's `retained_in_context` signal — emitted by the consolidation compiler's actual selection — is a new class of reward signal: **compiler-as-verifier**. This is our paper-length contribution.
2. **No paper learns a memory policy without rollouts.** Every prior work requires generating trajectories to train on. Morpheus's batch-nightly update on historical feedback with no counterfactual rollouts is unusual — it's closer to contextual-bandit or offline-RL territory that nobody has explored for memory.
3. **No paper crosses observation-level and digest-level scoring.** Morpheus's decision to also score `is_consolidated_digest` rows with the same bucket-weight machinery (Phase C) is unprecedented. HiGMem has hierarchical retrieval but unified retrieval scoring, not unified *policy* scoring.
4. **No public benchmark measures "did the right observation get injected" as an outcome metric.** RAG-Gym, PRMBench, RewardBench all score task-terminal outputs. Morpheus could define the first retention-quality benchmark tied to `retained_in_context` rate deltas.
5. **Calendar-delayed reward.** All RL-memory papers assume reward arrives within the same trajectory. Morpheus's feedback trickles in over days/weeks. No paper models calendar-time-discounted reward for memory; this is a genuine unsolved problem.

## Reading order for a new Morpheus engineer

1. **`morpheus-rl-design.md`** + **`morpheus-architecture.md`** — the brief.
2. **MEM1 (`2506.15841`)** — closest structural analog; read in full.
3. **MemAgent (`2507.02259`)** — the overwrite-strategy + Multi-Conv DAPO algorithm.
4. **Process vs. Outcome Reward (`2505.14069`)** — settles the "why process rewards" question.

Optional follow-ups: StepSearch (`2505.15107`) for IG/redundancy reward design, ReSum (`2509.13313`) for consolidation-job patterns, and the new `2604.18002` NGC paper for the forget-as-RL-action framing.
